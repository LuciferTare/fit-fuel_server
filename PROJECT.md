# Fit & Fuel — Backend Project Documentation

## Overview

Fit & Fuel is a Django REST Framework backend that powers a Flutter fitness-management mobile application. The backend manages a multi-tenant gym ecosystem with four distinct user roles — Platform Admin, Gym Owner, Trainer, and Member — each with strictly scoped permissions and API surfaces.

The system migrates the Flutter app from local SQLite storage to a centralized production backend backed by MySQL.

## Domain Model

### User Role Hierarchy

There is **one database table** (`users`) for all user types. Roles are distinguished by the `user_type` field. No separate Gym or Trainer models exist — gym identity is the UUID of the `GYM_OWNER` user.

```
ADMIN
  └── creates GYM_OWNERs
        └── creates TRAINERs (scoped to their gym)
        └── creates MEMBERs  (scoped to their gym)
              └── can be assigned to a TRAINER
```

### CustomUser Model

**Table:** `users`  
**Primary Key:** `uuid` (UUID4, not auto-int)  
**Auth identifier:** `phone_number` (`USERNAME_FIELD = "phone_number"`)

| Field               | Type           | Notes                                                             |
| ------------------- | -------------- | ----------------------------------------------------------------- |
| `uuid`              | UUIDField      | PK, auto-generated, not editable                                  |
| `phone_number`      | CharField(15)  | Unique — used as login username                                   |
| `first_name`        | CharField(150) | Optional                                                          |
| `last_name`         | CharField(150) | Optional                                                          |
| `date_of_birth`     | DateField      | Optional                                                          |
| `gender`            | CharField      | Choices: MALE / FEMALE / OTHER                                    |
| `profile_picture`   | ImageField     | Uploaded to `profile_pictures/`                                   |
| `user_type`         | CharField      | ADMIN / GYM_OWNER / TRAINER / MEMBER                              |
| `status`            | CharField      | ACTIVE / DISABLED / SUSPENDED / DELETED                           |
| `gym`               | FK → self      | Points to a `GYM_OWNER` user; null for Admin/GymOwner             |
| `trainer`           | FK → self      | Points to a `TRAINER` user; only set on MEMBERs                   |
| `trainer_limit`     | IntegerField   | Max trainers allowed for this Gym Owner (0 = unlimited)           |
| `membership_start`  | DateField      | Snapshot: start date of current/last membership (nullable)        |
| `membership_end`    | DateField      | Snapshot: end date of current/last membership (nullable, indexed) |
| `membership_status` | CharField      | Snapshot: ACTIVE / EXPIRED / null                                 |
| `membership_plan`   | CharField(100) | Snapshot: plan label from last payment (nullable)                 |
| `is_staff`          | BooleanField   | Django admin access flag                                          |
| `is_active`         | BooleanField   | Django auth flag                                                  |
| `created_by`        | FK → self      | Audit: who created this user                                      |
| `updated_by`        | FK → self      | Audit: who last updated this user                                 |
| `created_at`        | DateTimeField  | Auto set on create, indexed                                       |
| `updated_at`        | DateTimeField  | Auto set on update                                                |
| `is_deleted`        | BooleanField   | Soft-delete flag, indexed                                         |
| `deleted_at`        | DateTimeField  | Timestamp of soft deletion                                        |

**Indexes:** `db_index=True` on `user_type`, `status`, `created_at`, `is_deleted`, `membership_end`. Composite index `(gym, user_type)` for trainer/member queries.

**Membership snapshot fields** (`membership_start`, `membership_end`, `membership_status`, `membership_plan`) are denormalised copies updated whenever a payment is recorded via `POST /api/payments/`. They power the membership-expiry report without joining to the `memberships` table.

**Computed property:**

- `age` — calculated from `date_of_birth` relative to today's date.

**Methods:**

- `get_full_name()` — returns `"first_name last_name"` or falls back to `phone_number`
- `get_short_name()` — returns `first_name` or falls back to `phone_number`
- `soft_delete(deleted_by=None)` — sets `is_deleted=True`, `status=DELETED`, stamps `deleted_at`

**Relationship validation** (enforced in `clean()` / `_validate_relationships()`):

- ADMIN cannot have a gym or trainer assignment
- GYM_OWNER cannot have a gym or trainer assignment
- TRAINER cannot reference another trainer
- MEMBER's trainer must belong to the same gym as the member

### Membership Model

**Table:** `memberships`  
**Primary Key:** `uuid` (inherited from `BaseModel`)  
Tracks one membership term for a Member. A Member may accumulate multiple records over time (renewals). Only one `ACTIVE` membership with non-overlapping dates is enforced per Member.

| Field                       | Type               | Notes                                       |
| --------------------------- | ------------------ | ------------------------------------------- |
| `uuid`                      | UUIDField          | PK, auto-generated                          |
| `member`                    | FK → CustomUser    | Must be `user_type=MEMBER`                  |
| `start_date`                | DateField          | Inclusive start; must be ≤ `end_date`       |
| `end_date`                  | DateField          | Inclusive end                               |
| `plan`                      | CharField(100)     | Optional label, e.g. "Monthly", "Quarterly" |
| `amount_paid`               | DecimalField(10,2) | Total amount paid for this term             |
| `payment_mode`              | CharField          | CASH / ONLINE                               |
| `status`                    | CharField          | ACTIVE / EXPIRED (default: ACTIVE)          |
| `created_by` / `updated_by` | FK → User          | Audit (from BaseModel)                      |
| `created_at` / `updated_at` | DateTimeField      | Auto-managed                                |
| `is_deleted` / `deleted_at` | —                  | Soft-delete (from BaseModel)                |

### Payment Model

**Table:** `payments`  
**Primary Key:** `uuid` (inherited from `BaseModel`)  
Records an individual payment transaction against a Membership term. A membership may have multiple payments (e.g. installments).

| Field                       | Type               | Notes                                      |
| --------------------------- | ------------------ | ------------------------------------------ |
| `uuid`                      | UUIDField          | PK, auto-generated                         |
| `membership`                | FK → Membership    | The membership this payment covers         |
| `amount`                    | DecimalField(10,2) | Must be > 0                                |
| `mode`                      | CharField          | CASH / ONLINE                              |
| `paid_on`                   | DateTimeField      | Defaults to `timezone.now` if not provided |
| `created_by` / `updated_by` | FK → User          | Audit (from BaseModel)                     |
| `created_at` / `updated_at` | DateTimeField      | Auto-managed                               |
| `is_deleted` / `deleted_at` | —                  | Soft-delete (from BaseModel)               |

### Attendance Model

**Table:** `attendance`  
**Primary Key:** `uuid` (inherited from `BaseModel`)  
Records a single gym visit for a Member. Created on check-in; `check_out` and GPS coordinates are set on check-out.

| Field                       | Type              | Notes                                 |
| --------------------------- | ----------------- | ------------------------------------- |
| `uuid`                      | UUIDField         | PK, auto-generated                    |
| `member`                    | FK → CustomUser   | Must be `user_type=MEMBER`            |
| `check_in`                  | DateTimeField     | Required; set at check-in time        |
| `check_out`                 | DateTimeField     | Nullable; set at check-out time       |
| `check_in_lat`              | DecimalField(9,6) | GPS latitude at check-in (nullable)   |
| `check_in_lng`              | DecimalField(9,6) | GPS longitude at check-in (nullable)  |
| `check_out_lat`             | DecimalField(9,6) | GPS latitude at check-out (nullable)  |
| `check_out_lng`             | DecimalField(9,6) | GPS longitude at check-out (nullable) |
| `created_by` / `updated_by` | FK → User         | Audit (from BaseModel)                |
| `created_at` / `updated_at` | DateTimeField     | Auto-managed                          |
| `is_deleted` / `deleted_at` | —                 | Soft-delete (from BaseModel)          |

**Indexes:** Composite index `(member, check_in)` for inactivity queries.

## Authentication

### Strategy

- **No public registration.** Accounts are created by higher-privileged roles only.
- Login uses `phone_number` + `password`.
- JWT is issued on successful login; all other endpoints require `Bearer <token>` in the `Authorization` header.
- JWT payload carries `user_type` and `status` as custom claims.

### JWT Configuration

| Setting                  | Value                                                               |
| ------------------------ | ------------------------------------------------------------------- |
| Access token lifetime    | 1440 min (1 day) — configurable via `ACCESS_TOKEN_TIME` env var     |
| Refresh token lifetime   | 43200 min (30 days) — configurable via `REFRESH_TOKEN_TIME` env var |
| Algorithm                | HS256                                                               |
| Signing key              | `JWT_SECRET_KEY` env var                                            |
| User ID field            | `uuid` (not `id`)                                                   |
| User ID claim            | `user_id`                                                           |
| Rotate refresh tokens    | Yes                                                                 |
| Blacklist after rotation | Yes                                                                 |
| Update last login        | Yes                                                                 |
| Custom obtain serializer | `accounts.serializers.LoginSerializer`                              |

### Account Status Enforcement on Login

Before issuing tokens, `LoginSerializer` pre-checks account status and raises HTTP 403 for:

- `DELETED` accounts
- `DISABLED` accounts
- `SUSPENDED` accounts

Invalid credentials raise HTTP 401.

### Token Blacklist

On logout the refresh token is explicitly blacklisted via `RefreshToken.blacklist()`. Token rotation also blacklists the old refresh token automatically.

---

## Password Policy

Enforced by `accounts.validators.StrongPasswordValidator` (registered in `AUTH_PASSWORD_VALIDATORS`) and mirrored inline in serializers via `_validate_password_strength()`:

- Minimum 8 characters
- At least one uppercase letter
- At least one lowercase letter
- At least one digit
- At least one special character from `!@#$%^&*(),.?":{}|<>-_=+[]\;'\`~/`

## API Endpoints

### Base URL Pattern

All responses are wrapped in the `ResponseRenderer` envelope:

```json
{
  "data": <payload or null>,
  "message": "<string>",
  "status": <http_status_code>,
  "time": "<ISO datetime>"
}
```

For paginated list responses the envelope also includes `count`, `next`, `previous`.

### Phase-3 Member Payments — `/api/payments/`

**Permission:** `IsAdminOrGymOwner`  
A single-step endpoint that records a payment **and** updates the member's membership snapshot fields (`membership_start`, `membership_end`, `membership_status`) in one atomic operation. Internally creates a `Membership` record.

| Method | Path             | Description                                                              |
| ------ | ---------------- | ------------------------------------------------------------------------ |
| POST   | `/api/payments/` | Record a membership payment; update member's membership dates and status |
| GET    | `/api/payments/` | List memberships for the gym; filter by `?member_id=`                    |

**POST body:**

```json
{
  "member_id": "<uuid>",
  "date": "2026-06-01",
  "amount": "1500.00",
  "mode": "Cash",
  "start_date": "2026-06-01",
  "end_date": "2026-06-30",
  "plan": "Monthly"
}
```

> **Note:** The legacy `/payments/` endpoint (Phase 2) remains at the root level and records individual payment transactions against an existing `Membership` FK. `/api/payments/` is the simpler Phase-3 interface.

### Backup / Sync — `/api/backup/`

**Permission:** `IsAdmin | IsGymOwner | IsTrainer`  
Offline-first sync endpoints. The server acts as a cloud backup; conflict resolution uses Last-Write-Wins (LWW) on `updated_at`.

| Method | Path                    | Description                                                                                       |
| ------ | ----------------------- | ------------------------------------------------------------------------------------------------- |
| POST   | `/api/backup/upload/`   | Push client-side changes to the server (supports `create`, `update`, `delete` actions per record) |
| GET    | `/api/backup/download/` | Pull server-side records changed since a timestamp (`?user_id=&since=<ISO datetime>`)             |

**Upload payload:**

```json
{
  "user_id": "<member uuid>",
  "changes": [
    {"model": "attendance.Attendance", "action": "create", "data": {...}},
    {"model": "attendance.Attendance", "action": "update", "data": {...}},
    {"model": "attendance.Attendance", "action": "delete", "data": {"uuid": "..."}}
  ]
}
```

**LWW rule:** On `update`, if the server record's `updated_at ≥` client's `updated_at`, the change is skipped (server wins). The response summarises `created`, `updated`, `deleted`, `skipped`, `errors` counts.  
**Supported models:** `attendance.Attendance` (Phase 3). Extend `_SYNCABLE` dict in `backup/views.py` for Phase 4 workout models.

## Soft Delete

All user deletions via the API are **soft deletes** — no records are physically removed.

`soft_delete(deleted_by=None)` sets:

- `is_deleted = True`
- `status = DELETED`
- `deleted_at = now()`
- `updated_by = deleted_by` (if provided)

The `ActiveUserManager` (`active_objects`) automatically excludes soft-deleted records. All view querysets use `active_objects`, so deleted users are invisible to the API.

## Response Envelope (`ResponseRenderer`)

Every API response is normalized through `core.renderers.ResponseRenderer`:

```json
{
  "data": <payload>,
  "message": "<string>",
  "status": <http_code>,
  "time": "<datetime>"
}
```

- On **2xx**: `data` contains the payload; `message` is the `detail` field if present; for paginated responses `count`, `next`, `previous` are hoisted to the top level.
- On **non-2xx**: `data` is set to `null`; `message` is populated from `detail`.

## Data Scoping Rules

| Actor     | Visible Data                                                                                                        |
| --------- | ------------------------------------------------------------------------------------------------------------------- |
| Admin     | All Gym Owners, Trainers, Members, Memberships, Payments, Attendance (across all gyms)                              |
| Gym Owner | Trainers and Members in their gym; Memberships, Payments, Attendance for their members; Reports scoped to their gym |
| Trainer   | Only Members assigned to them (`trainer=request.user`); Attendance for their assigned members                       |
| Member    | Only their own profile (no server endpoints for attendance/payments — all offline)                                  |

Cross-gym operations (e.g., assigning a trainer from a different gym) are rejected at the queryset level, not just in business logic — the foreign-gym trainer simply does not appear in the queryset, resulting in HTTP 404.

Membership/Payment scoping is enforced via queryset filters (`member__gym=request.user` and `membership__member__gym=request.user` respectively).

Attendance scoping: `member__gym=request.user` for Gym Owners; `member__trainer=request.user` for Trainers.
